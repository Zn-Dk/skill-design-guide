# Pipeline 设计模式

> 📌 **适用范围：Heavy（Pipeline 型）Skill**
>
> 本章节仅适用于需要多阶段数据处理、脚本编排、状态传递的重型 Skill。
> Lite 和 Standard 档 Skill 无需关注此章节。

---

## 3.1 阶段划分原则

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ P1 解析   │──▶│ P2 配置   │──▶│ P3 流量   │──▶│ P4 合成   │──▶│ P5 推理   │──▶│ P6 报告   │
│  (脚本)   │   │  (脚本)   │   │  (脚本)   │   │  (脚本)   │   │  (LLM)   │   │(脚本+审计)│
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

**划分依据**：

1. **输入归一化**（P1）：多入口统一为结构化标识
2. **数据采集**（P2-P3）：从外部系统拉取原始数据，一类数据源一个阶段
3. **结构化建模**（P4）：将采集数据合成为领域模型（如图模型 nodes+edges）
4. **LLM 推理**（P5）：唯一需要"理解"的阶段——模式识别、归因、表达
5. **报告生成+审计**（P6）：模板渲染 + 规则校验，闭环质量保障

---

## 3.2 阶段间耦合规则

| 规则 | 说明 |
|------|------|
| 单向依赖 | 后阶段读取前阶段的 state 字段，禁止反向修改 |
| State Merge | 每阶段完成后通过 `state_manager.py --merge` 合入，保证原子性 |
| 幂等重跑 | 任何阶段可单独重跑，结果覆盖 state 中对应字段 |
| 审计闭环 | 最终审计失败 → 回溯重跑对应阶段，禁止绕过 |

---

## 3.3 LLM 阶段的工程化保障

LLM 自由推理最容易出错，需要以下保障：

| 保障机制 | 实现方式 |
|---------|---------|
| Few-shot 示例 | `phases/05_examples/` 按类型提供 JSON 格式的完整示例 |
| 输出 Schema 约束 | `schemas/state_schema.json` 中严格定义 P5 输出结构 |
| Checklist 驱动 | Phase 指引中列出"必须完成的 N 个子任务" |
| 格式模板 | 每个输出字段给出 JSON 格式样例，禁止自创字段名 |
| 参考知识注入 | 指明需要读取的 knowledge/ 文件 |

---

## 设计要点总结

- **脚本阶段**：确定性、可重复、可审计
- **LLM 阶段**：约束输出格式、提供示例、限定知识范围
- **阶段间**：通过 state.json 单向传递，保证幂等性和可追溯性
