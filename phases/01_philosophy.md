# 核心设计哲学

---

## 1.1 最小必要原则

> ⚠️ **铁律**：不要为了"完整"而创建空目录或空文件。每个文件必须有实际内容才创建。

| 维度 | 原则 | 反模式 |
|------|------|--------|
| 目录结构 | 按执行模式选择对应档位，不多不少 | 所有 Skill 都套用六层完整结构 |
| 文档内容 | 每行文字都应对 Agent 决策有直接帮助 | 大段解释性文字、重复说明 |
| 工具声明 | 只声明实际需要的工具权限 | `allowed-tools: [*]` |
| 子模块 | 只在内容超过 SKILL.md 承载能力时拆分 | 为了"模块化"而拆出 3 行内容的文件 |

**判断标准**：如果删除某个文件/目录后 Skill 功能不受影响，那它就不应该存在。

## 1.2 确定性优先

> ⚠️ **铁律**：能用代码/模式/脚本做的，绝不交给 LLM 自然语言理解。

| 层次 | 执行方式 | 适用场景 |
|------|---------|---------|
| 模式匹配 | **grep/regex 模式** | 代码规范检查、命名规则校验 |
| 数据处理 | **脚本（Python/Shell）** | API 调用、数据清洗、格式转换 |
| 模板填充 | **模板 + 占位符** | 文档生成、代码脚手架 |
| 语义理解 | **LLM Agent** | 推理、归因、创意表达——必须理解语义才能完成 |

**原则**：LLM 只做"必须理解语义才能完成"的任务。确定性操作交给确定性工具。

**示例对比**：

```markdown
## ❌ 低效：纯自然语言规则
"检查代码中是否有未使用 @/utils/validate 中 isUrl 函数的手写 URL 验证"

## ✅ 高效：grep 模式 + 判定规则
- 检测模式：`/https?:\/\/|new URL\(|\.match\(.*url/i`
- 排除路径：`src/utils/validate.ts`
- 判定：匹配且不在排除路径 → 违规
- 修复：`import { isUrl } from '@/utils/validate'`
```

## 1.3 Token 预算意识

> ⚠️ **铁律**：Skill 的本质是在有限上下文窗口中，以最少 token 传递最精准的信息。

| 原则 | 说明 | 预算参考 |
|------|------|---------|
| 入口精简 | SKILL.md 是 Agent 首先加载的内容，必须精炼 | ≤ 2KB（约 500 token） |
| 按需加载 | 子文档只在 Agent 判断需要时才读取 | 单个子文档 ≤ 4KB |
| 信息密度 | 删除一切"解释性废话"和重复内容 | 每行都应影响 Agent 决策 |
| 渐进式披露 | 分层组织，80% 决策在 L1 完成 | 见 `10_performance.md` |

**自检问题**：
- 这段文字删掉后，Agent 的行为会变差吗？如果不会，删掉。
- 这个信息 Agent 每次都需要吗？如果不是，下沉到子文档。
- 这个规则能用代码/模式表达吗？如果能，不要用自然语言。

## 附：Heavy 档专用原则

以下原则**仅适用于 Heavy（Pipeline 型）Skill**，不作为通用要求：

### State-First（状态优先）

所有阶段的输入输出通过统一的 `state.json` 文件串联：
- 每个 phase 执行后必须 merge 回 state
- state 是唯一 SSOT，禁止阶段间靠 LLM 记忆传递信息
- state 有严格 JSON Schema 定义结构约束

### 证据链驱动

每条结论必须附带可追溯的证据链接（监控 URL / 配置平台 URL），审计时逐条校验。

> 📌 这两个原则的详细说明见 `03_pipeline.md`。
