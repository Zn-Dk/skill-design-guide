# 质量保障体系

> 📌 **通用章节**：所有档位 Skill 均需关注质量保障，但具体机制按档位分级。

---

## 通用质量原则（全档位适用）

### 1. 可验证性

每个 Skill 的产出必须可验证：

| 档位 | 验证方式 |
|------|---------|
| **Lite** | 人工审阅 + 正/反例对比 |
| **Standard** | grep 模式匹配 + 模板完整性检查 |
| **Heavy** | Schema 校验 + 审计脚本 + 自动化测试 |

### 2. 确定性优先

- 能用代码/模式/脚本验证的，不交给 LLM 自然语言判断
- 规则越确定，质量越可控
- 模糊规则必须附带正/反例，减少歧义

### 3. 渐进式质量投入

不要一开始就追求完美的质量体系。按以下顺序逐步建设：

```
Step 1: 定义"什么是正确的产出"（正/反例）
Step 2: 将判断规则代码化（grep 模式 / Schema）
Step 3: 自动化校验（审计脚本）
Step 4: 闭环反馈（用户纠正 → 规则更新）
```

### 4. 错误分级

所有档位都应区分错误严重度：

| 严重度 | 行为 | 示例 |
|--------|------|------|
| **ERROR** | 阻断产出，必须修正 | 关键字段缺失、逻辑矛盾 |
| **WARNING** | 允许产出但标记风险 | 非关键信息缺失、格式瑕疵 |
| **INFO** | 仅记录，不影响产出 | 优化建议、风格偏好 |

---

## Lite 档质量保障

**核心机制**：正/反例 + 人工审阅

- SKILL.md 中每条规则必须附带至少一个正例和一个反例
- 正/反例本身就是质量标准的定义
- 定期审阅：规则是否仍然适用？是否有新的边界 case？

---

## Standard 档质量保障

**核心机制**：模式匹配 + 模板完整性

- 审查类规则：提供 grep 检测模式，可自动化验证
- 生成类规则：提供模板 + 必填占位符清单，检查产出完整性
- 建立 `assets/examples/` 目录，存放"标准产出"作为参照

**自检清单**：
- [ ] 每条审查规则都有对应的检测模式？
- [ ] 每个模板的必填项都有标注？
- [ ] 有至少一个完整的"标准产出"示例？

## Heavy 档质量保障（三道防线）

> 📌 **适用范围**：仅 Heavy（Pipeline 型）Skill 需要完整三道防线。

### 第一道防线：Schema 约束

**时机**：每次脚本输出数据、每次 state merge 时自动触发。

**实现方式**：
- 使用 `schemas/` 目录下的 JSON Schema 定义每个阶段的输出结构
- `state_manager.py --merge` 时自动校验新数据是否符合 schema
- 校验失败 → 立即报错，阻止脏数据进入 state

**设计要点**：
- Schema 应覆盖所有必填字段和类型约束
- 对枚举值使用 `enum` 约束，避免 LLM 自创值
- 对数组长度、字符串格式等使用 `minItems`、`pattern` 等约束

### 第二道防线：审计脚本

**时机**：报告生成阶段，对最终产物进行规则校验。

**实现方式**：
- `scripts/auditor.py` 读取完整 state + 报告产物
- 按预定义规则逐条校验
- 输出审计报告（通过/警告/错误）

**关键原则**：
- 审计不可绕过：ERROR 级别必须修正根因，禁止手动跳过
- 审计规则应随 `.learnings/CORRECTIONS.md` 的积累持续补充
- 审计脚本本身也需要版本管理和测试

### 第三道防线：自反哺

**时机**：用户使用产物后给出反馈时触发。

**闭环流程**：
```
用户反馈 → 记录到 CORRECTIONS.md → 分析根因 → 更新知识库规则
                                                    ↓
                                    下次执行时自动应用新规则 → 产出更准确
```

## 实践建议

| 档位 | 起步建议 |
|------|---------|
| **Lite** | 写好正/反例即可，不需要额外质量机制 |
| **Standard** | 先写检测模式，再补标准产出示例 |
| **Heavy** | 先定义 Schema → 再写审计规则 → 最后建立反馈闭环 |

**通用原则**：
- 避免过度约束：质量机制应捕捉"明确的错误"，而非限制 LLM 的合理创造性
- 质量投入与 Skill 复杂度成正比：简单 Skill 不需要重型质量体系
- 测试你的质量机制：用已知的"好产物"和"坏产物"验证规则准确性
