# 10. 性能与披露策略

> Skill 的本质是在有限的上下文窗口中，以最少的 Token 传递最精准的信息。

---

## 核心原则

| 原则 | 说明 |
|------|------|
| **Token 预算意识** | 入口 SKILL.md ≤ 2KB（约 500 token），Agent 在 L1 应能完成 80% 决策 |
| **按需加载** | 子文档只在 Agent 判断需要时才读取，不预加载全部内容 |
| **信息密度** | 每行文字都应对 Agent 的决策有直接帮助，删除一切"解释性废话" |
| **确定性优先** | 能用代码/模式/脚本做的，绝不交给 LLM 自然语言理解 |

---

## 渐进式披露层级

```
Level 0: SKILL.md frontmatter
         → Agent 判断是否需要此 Skill
         → 预算: `description` ≤ 250 字符（含换行）

Level 1: SKILL.md 正文
         → 核心原则 + 模块索引表
         → 预算: < 500 token
         → 目标: 80% 的决策在此完成

Level 2: phases/*.md / modules/*.md
         → 具体规则、模板、检测模式
         → 按需加载单个文件
         → 目标: 处理具体执行细节

Level 3: assets/*.md / reference/*.md
         → 完整参考资料、边界 case 说明
         → 极少加载
         → 目标: 仅在遇到罕见情况时使用
```

**铁律**：Agent 在 Level 1 就应该能完成 80% 的决策。只有 20% 的边界 case 需要深入到 Level 2-3。

### Frontmatter `description` 压缩规则

- 只保留加载判断所需的最小信息：**是什么 / 覆盖什么 / 何时使用**
- 优先写成 **2-3 行短句**，避免超过 **250 字符**
- 子模块名、完整流程、长场景枚举不要放在 `description`
- 如果一句话删掉后不影响加载判断，就应删除

---

## 不同场景的性能策略

| Skill 场景 | 性能瓶颈 | 优化策略 |
|------------|---------|---------|
| **Review Skill** | Agent 用自然语言"理解"规则太慢 | 用 grep 模式 + 判定表替代自然语言规则 |
| **生成 Skill** | Agent 从零构思结构太慢 | 提供完整模板 + 占位符，Agent 只做填充 |
| **分析 Skill** | Agent 处理大量数据太慢 | 脚本预处理 → 只给 Agent 结构化摘要 |
| **指引 Skill** | 规则太多 Agent 记不住 | 分层：核心规则 ≤ 7 条，详细规则按需加载 |

## 信息密度检查清单

编写 Skill 文档时，逐行自检：

- [ ] **这行删掉后，Agent 的行为会变差吗？** — 如果不会，删掉
- [ ] **这段话能用表格/列表替代吗？** — 表格比段落信息密度高 3-5 倍
- [ ] **这个规则能用代码模式表达吗？** — grep 模式比自然语言描述精确 10 倍
- [ ] **这个示例是必要的吗？** — 只保留能消除歧义的示例，删除"锦上添花"的示例
- [ ] **frontmatter `description` 还能再删吗？** — 只保留加载判断必需信息，控制在 250 字符内

### 反模式示例

```markdown
## ❌ 低密度写法（浪费 Token）
在进行代码审查时，我们需要特别注意一些常见的问题。
首先，我们应该检查是否有未使用的导入语句，因为这些语句
虽然不会影响运行时性能，但会增加代码的认知负担，
让其他开发者在阅读代码时感到困惑...

## ✅ 高密度写法（精准高效）
### 规则：未使用导入
- **检测**：`/^import .+ from .+/ → 检查引用计数 = 0`
- **判定**：导入存在但无引用 → 违规
- **修复**：删除该行
```

## SKILL.md 正文的写作模板

一个高性能的 SKILL.md 正文应该遵循以下结构：

```markdown
# {Skill 名称}

{一句话描述核心职责}

## 核心原则（≤ 7 条）

1. **{原则名}** — {一句话说明}
2. ...

## 模块索引

| 文件 | 职责 | 何时加载 |
|------|------|---------|
| `phases/xx.md` | {职责} | {触发条件} |
| ... | ... | ... |

## 快速决策树（可选）

- 如果 {条件A} → 执行 {动作A}
- 如果 {条件B} → 加载 `phases/xx.md` 获取详细规则
- 否则 → {默认动作}
```

## Token 预算分配参考

| 档位 | SKILL.md | 单个子文档 | 总预算上限 |
|------|---------|-----------|-----------|
| **Lite** | ≤ 2KB | — | 2KB |
| **Standard** | ≤ 2KB | ≤ 4KB/个 | 20KB |
| **Heavy** | ≤ 2KB | ≤ 6KB/个 | 50KB |

> ⚠️ 这些是建议值，不是硬限制。但超出时应该反思：是否信息密度不够？是否可以拆分？
