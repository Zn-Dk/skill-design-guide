# 多实例并行（Team 模式）

> 📌 **高级特性**：仅 Heavy（Pipeline 型）Skill 需要。大多数 Skill 不涉及多实例并行，可跳过本章。

当需要对多个输入并行执行同一工作流时，采用 Team 模式编排。

---

## 架构概览

```
主 Agent
  ├── 创建 Team
  ├── spawn sub-agent-1（输入 A，独立执行 P1→P6）
  ├── spawn sub-agent-2（输入 B，独立执行 P1→P6）
  └── 汇总审计
       ├── 校验每个产物完整性
       ├── 运行 auditor 校验
       ├── 补偿失败场景（最多重试 1 次）
       └── 生成批量汇总
```

---

## 关键设计原则

| 原则 | 说明 |
|------|------|
| **独立 output_dir** | 每个 sub-agent 有独立的输出目录，互不干扰 |
| **无状态工作流** | 同一个 Skill 定义可以被多个 sub-agent 并发使用 |
| **主 Agent 只做编排** | 主 Agent 负责编排和审计，不参与具体分析 |
| **失败补偿** | 单个 sub-agent 失败不影响整体，最多重试 1 次 |

---

## 适用场景

- 批量处理多个独立目标（如多个服务、多个模块、多个数据源）
- 同一工作流需要对不同输入并行执行
- 需要汇总多个独立执行结果

---

## 实施要点

1. **输入隔离**：每个 sub-agent 接收独立的输入参数，不共享可变状态
2. **输出隔离**：每个 sub-agent 写入独立的 `output_dir`，避免文件冲突
3. **汇总审计**：主 Agent 在所有 sub-agent 完成后，统一校验和汇总
4. **错误处理**：单个失败不阻断整体流程，记录失败原因后继续

---

## 与单实例模式的对比

| 维度 | 单实例 | Team 模式 |
|------|--------|-----------|
| 输入 | 单个 | 多个并行 |
| 状态管理 | 单一 state.json | 每个 sub-agent 独立 state |
| 审计 | P6 阶段内完成 | 主 Agent 统一审计 |
| 适用规模 | 1 个分析目标 | N 个分析目标 |
