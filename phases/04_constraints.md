# 全局约束设计模式

> 📌 **适用范围**：所有档位（Lite / Standard / Heavy）

在 `SKILL.md` 中用 `> ⚠️` 格式定义**不可违反的铁律**：

```markdown
> ⚠️ **约束名**：具体规则描述
```

---

## 约束分类

| 类型 | 说明 | 示例 |
|------|------|------|
| 职责边界 | 明确 Skill 做什么、不做什么 | "只负责审查，不自动修复" |
| 格式约束 | 输入/输出的格式要求 | "报告格式必须是 Markdown" |
| 质量门禁 | 产出必须满足的最低标准 | "每条结论必须附可验证依据" |
| 安全约束 | 防止危险操作 | "禁止执行 rm -rf 等破坏性命令" |
| 独立性 | 减少外部依赖 | "脚本 HTTP 直连：不依赖 Agent 中转" |
| 状态管理 | 阶段间数据传递规则 | "每阶段必须 merge state，禁止跳过"（Heavy 型专用） |

---

## 设计原则

1. **约束必须可验证** — 每条约束应能通过脚本、grep 模式或审计规则自动检查
2. **约束必须有理由** — 不写"因为规范要求"，而是写"因为 X 场景下会导致 Y 问题"
3. **约束数量克制** — 铁律不超过 10 条，过多则失去约束力
4. **约束分级** — ERROR 级（阻断输出）vs WARNING 级（标记风险但允许产出）

---

## 不同档位的约束策略

| 档位 | 约束数量 | 验证方式 | 典型约束 |
|------|---------|---------|---------|
| **Lite** | 3-5 条 | 自然语言规则 + 正/反例 | 格式约束、职责边界 |
| **Standard** | 5-8 条 | grep 模式 + 判定规则 | 质量门禁、安全约束 |
| **Heavy** | 8-10 条 | 脚本自动审计 | 状态管理、证据完整性 |

---

## 约束放置位置

| 位置 | 作用域 | 说明 |
|------|--------|------|
| `SKILL.md` 顶层 | 全局 | 所有阶段/模块都必须遵守 |
| `phases/*.md` 或 `modules/*.md` 内部 | 局部 | 仅该阶段/模块需要遵守的特定约束 |
| `schemas/*.json`（Heavy 型） | 结构级 | 通过 JSON Schema 强制约束数据格式 |

---

## 编写模板

```markdown
> ⚠️ **{约束名}**：{一句话规则描述}
>
> 原因：{为什么需要这条约束}
> 验证：{如何检查是否违反}
> 级别：ERROR | WARNING
```

### 示例

```markdown
> ⚠️ **确定性优先**：能用 grep 模式/代码模板/脚本做的检查，绝不交给 LLM 自然语言理解
>
> 原因：自然语言理解存在不确定性，可能漏检或误判
> 验证：审查规则文档中是否存在可转化为 grep 模式但未转化的自然语言描述
> 级别：WARNING
```
