# 知识管理模式

> 📌 适用范围：Standard / Heavy 档（Lite 档通常将知识内联于 SKILL.md）

---

## SSOT（Single Source of Truth）原则

同一份知识只在一个位置维护，其他地方通过引用获取：

| 原则 | 说明 | 反模式 |
|------|------|--------|
| **唯一性** | 一条规则只写在一个文件中 | 同一规则在 SKILL.md 和 modules/ 中各写一遍 |
| **分层消费** | 脚本消费结构化数据，LLM 消费自然语言 | 让 LLM 解析 YAML 配置 |
| **版本可追溯** | 知识文件随 Git 管理，变更有据可查 | 口头约定、未落文档的隐式知识 |

---

## 按需加载策略

知识文件不应一次性全部加载到上下文，而是按 Agent 的决策需求逐步读取：

### 加载层级

| 层级 | 触发条件 | 加载内容 |
|------|---------|---------|
| L0 | Skill 被激活 | 仅 SKILL.md frontmatter + 正文索引 |
| L1 | Agent 确认需要某模块 | 对应的 phases/*.md 或 modules/*.md |
| L2 | 遇到具体规则/模板需求 | assets/ 中的模板或参考文件 |
| L3 | 边界 case / 深度排查 | reference/ 或 scripts/ |

### 设计要点

1. **入口轻量** — SKILL.md 只放索引和核心原则，不放具体规则细节
2. **文件自包含** — 每个子文件应能独立阅读，不依赖其他子文件的上下文
3. **交叉引用用链接** — 需要引用其他文件时，用相对路径链接而非复制内容

## 知识文件组织规范

### Standard 档

```
my-skill/
├── SKILL.md              # 入口：核心原则 + 模块索引
├── modules/
│   ├── rule-a.md         # 规则模块 A（自包含）
│   └── rule-b.md         # 规则模块 B（自包含）
└── assets/
    └── template.md       # 模板/参考资料
```

### Heavy 档

```
my-skill/
├── SKILL.md
├── phases/               # 阶段指引
├── assets/               # 模板 + 参考
├── scripts/              # 自动化脚本
└── schemas/              # 结构化数据定义
```

### 命名原则

- 文件名：`{功能描述}.{格式}`，全小写 + 连字符分隔（kebab-case）
- 目录名：复数形式（`modules/`、`assets/`、`scripts/`）
- 避免深层嵌套：最多 2 层目录深度

## 知识更新机制

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| **手动维护** | 规则变更频率低 | 人工编辑 + Git commit |
| **自反哺** | Skill 执行中发现新模式 | Agent 提议 → 人工审核 → 合入 |
| **脚本生成** | 可从代码库自动提取的知识 | 定期运行脚本更新（Heavy 档专用） |

> ⚠️ 无论哪种策略，知识变更都必须经过人工审核后才能合入。
